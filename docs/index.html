El problema

¿Por qué?

El caso de uso: arrancar un SaaS con SQLite o libSQL —cero operación, sin servidor de base de datos que administrar— y migrar a PostgreSQL más adelante, cuando el crecimiento lo exija, sin apagar la aplicación.

Cada fase —auditoría de capacidades, snapshot, replicación, cutover— produce un veredicto verificable (exit code y JSON) en vez de una promesa optimista. La compatibilidad la certifican pruebas contra motores reales, no la confianza.

Capacidades

Qué hace

Contrato de esquema canónico

Declara motor, versión y capacidades requeridas. La auditoría devuelve un veredicto por capacidad (exact / unknown) y corta la migración ante cualquier cosa no garantizada.

Snapshot verificado por digest

Exporta esquema y datos a una forma canónica, los importa en el destino y compara hashes canónicos de ambos lados antes de dar la copia por buena.

Replicación incremental por triggers

Captura automática de INSERT/UPDATE/DELETE, aplicación idempotente por secuencia, detección de conflictos y supresión de ecos transaccional bajo MVCC.

Cutover sin ventana

Audita, instala captura, hace snapshot, drena el journal y verifica equivalencia. Incluye un modo --dry-run de sólo lectura que imprime el plan sin escribir nada.

Tipo vector(N) de primera clase

SQLite/libSQL (F32_BLOB(N)) ↔ PostgreSQL (pgvector), validado end-to-end contra motores reales, incluida inspección de dimensión, snapshot y replicación.

Diseñado para agentes de IA

Un único binario compat con salidas JSON parseables y una taxonomía cerrada de códigos de error tipados: un agente ramifica por código, sin parsear texto libre.

Diseño

Arquitectura

Una representación canónica se coloca entre los dos motores. Ningún componente considera una transformación exacta si no puede reconstruir y comprobar su semántica en ambos lados.

SQLite/libSQL → Inspección → AST canónico → Auditoría → (exact) Compilador DDL → PostgreSQL / (unknown) Detener. Snapshot y Journal → Replicación → Verificación.

Proceso

Ciclo de migración

Cuatro fases, cada una con su veredicto determinista. Un fallo detiene el proceso con un código tipado; el corte final del DSN de la aplicación es manual, tras recibir status=ready.

1 Audit → 2 Snapshot → 3 Replicate → 4 Cutover → status=ready → corte manual del DSN.

El sistema no declara compatibilidad total: la aceptación global sigue fallando mientras exista una familia genérica requerida con estado distinto de exact. Es una decisión de honestidad, no un defecto.

Gramática canónica

El subconjunto de SQL que se traduce y verifica en ambos motores — acotado a propósito y en crecimiento. Todo lo que queda fuera se rechaza con un error explícito, nunca se degrada en silencio.

Interfaz

La interfaz de línea de comandos

Un único binario compat despacha tres subcomandos. Cada operación devuelve JSON legible por máquina, un envelope de error tipado y un exit code estable.

compat audit <contract.json>

Audita un contrato y evalúa cada capacidad requerida. Exit 0 si todo es exact; 2 ante uso incorrecto.

compat copy <migration.json>

Migración por snapshot: audita, exporta el origen, importa en el destino y verifica digests.

compat cutover [--dry-run] <cutover.json>

Cutover sin ventana: audita, instala captura, hace snapshot, drena el journal y verifica. --dry-run sólo audita y prueba conectividad.

compat

Sin subcomando, con uno desconocido o con --help: uso en stderr y un envelope ERR_USAGE (exit 2).

$ compat audit ./contract.json
[{"feature":"canonical_foreign_keys","status":"exact"},
 {"feature":"canonical_triggers","status":"exact"}]
# exit 0

$ compat copy ./migration.json   # destino divergente
{"status":"error","code":"ERR_VERIFY_DIVERGED",
 "message":"snapshot mismatch: source 7fcf9d… destination 7369f5…"}
# exit 1

Taxonomía de errores

Toda salida es JSON parseable, incluidos los errores. Un fallo emite un envelope { status, code, message } sobre esta taxonomía cerrada.

ERR_USAGE ERR_CONFIG ERR_AUDIT_NOT_EXACT ERR_CONNECT_SOURCE ERR_CONNECT_DESTINATION ERR_SCHEMA ERR_SNAPSHOT ERR_REPLICATION_CONFLICT ERR_CAPTURE ERR_VERIFY_DIVERGED ERR_INTERNAL
Empezar

Inicio rápido

Requiere Go. Cloná el repositorio y ejecutá:

1

Cloná y verificá que compila y pasa las pruebas:

git clone https://github.com/MauricioPerera/sqlite-postgres-compat
cd sqlite-postgres-compat
go test ./...  &&  go vet ./...
2

Auditá un contrato de ejemplo (no necesita bases de datos):

go run ./cmd/compat audit ./examples/contract.example.json
3

Probá un cutover en seco (editá examples/ con DSN reales primero):

go run ./cmd/compat cutover --dry-run ./examples/cutover.example.json
Para copiar un snapshot o hacer un cutover real, editá los ejemplos de examples/ con DSN reales primero. El contrato AGENTS.md documenta la gramática canónica machine-facing.
